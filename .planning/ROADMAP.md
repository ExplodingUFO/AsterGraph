# Roadmap: v0.38.0-beta External Adopter Feedback Loop

**Goal:** Turn the hardened `0.12.0-beta` authoring-surface line into an active feedback-collection and rapid-response loop. Gather real adopter signals through support-bundle intake, issue templates, and direct host evaluations; triage and fix friction points in documentation, samples, and the canonical route before committing to the next capability wave.

## Current Milestone

- **Version:** `v0.38.0-beta`
- **Name:** `External Adopter Feedback Loop`
- **Status:** active locally
- **Focus:** external adopter feedback intake, triage workflow, rapid polish, telemetry hardening
- **Phase numbering:** continues from `v0.37.0-beta` and starts at **Phase 281**

## Phases

- [ ] **Phase 281 Support Bundle Hardening And Issue Template Activation** — Make the support bundle capture enough context for first-line triage; activate the GitHub issue template for structured adopter reports.

- [ ] **Phase 284 Schema Version And Host Health Telemetry** — Harden support-bundle schema validation and public-tag mapping; add host health snapshot to proof markers for cross-environment comparison.

## Phase Details

### Phase 281 Support Bundle Hardening And Issue Template Activation

**Status:** planned

**Goal:** make the support bundle capture enough context for first-line triage; activate the GitHub issue template for structured adopter reports.

**Depends on:** Phase 280 (previous milestone)

**Requirements:** INTAKE-01, INTAKE-02

**Success criteria:**
1. A support bundle generated by any host contains environment, metrics, reproduction steps, and parameter snapshots sufficient for classification without follow-up questions.
2. The GitHub issue template actively asks for the support bundle and host integration context (framework version, adapter, custom node usage).
3. ConsumerSample proof tests validate that support bundles meet the new completeness criteria.

### Phase 282 Triage Workflow And Rapid Docs Fix

**Status:** planned

**Goal:** define intake criteria and triage routing; fix any docs/sample friction reported by ≥2 adopters with regression-proof markers.

**Depends on:** Phase 281

**Requirements:** INTAKE-03, POLISH-01

**Success criteria:**
1. There is a documented intake criteria checklist and a triage workflow that routes reports to docs-fix, sample-fix, or capability-gap buckets.
2. Any friction point reported by ≥2 adopters on the same docs page or sample route has a fix with a proof marker that fails on regression.
3. The triage workflow is referenced from CONTRIBUTING.md and the issue template.

### Phase 283 Ladder Validation And Onboarding Failure Detection

**Status:** planned

**Goal:** validate the hosted route ladder by external attempt; detect common onboarding failures explicitly in proof harness.

**Depends on:** Phase 282

**Requirements:** POLISH-02, POLISH-03

**Success criteria:**
1. The `Starter.Avalonia → HelloWorld.Avalonia → ConsumerSample.Avalonia` ladder is validated by at least one external attempt (simulated or real) and gaps are documented and fixed.
2. Common onboarding failures (missing package reference, wrong initialization order, overlooked compatibility service) are detected by the proof harness and surfaced as explicit error markers.
3. Onboarding failure markers are tested in ConsumerSample.Tests with regression-proof assertions.

### Phase 284 Schema Version And Host Health Telemetry

**Status:** planned

**Goal:** harden support-bundle schema validation and public-tag mapping; add host health snapshot to proof markers for cross-environment comparison.

**Depends on:** Phase 283

**Requirements:** TELEMETRY-01, TELEMETRY-02, TELEMETRY-03

**Success criteria:**
1. The support bundle includes a schema version and a public tag that map unambiguously to a released package version.
2. ConsumerSample proof markers include a "host health" snapshot (startup time, plugin scan, command latency) comparable across environments.
3. The support-bundle write path validates JSON schema before persisting and emits `SUPPORT_BUNDLE_SCHEMA_OK`; malformed bundles are rejected with a clear reason.

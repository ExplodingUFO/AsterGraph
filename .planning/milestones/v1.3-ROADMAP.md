# Milestone v1.3: Demo Showcase

**Status:** ✅ SHIPPED 2026-04-08
**Phases:** 19-21
**Total Plans:** 9

## Overview

AsterGraph turned `AsterGraph.Demo` into a graph-first, host-menu-first SDK showcase. The shipped demo now keeps one live graph session on screen while host-level menu groups expose shell, behavior, runtime, and proof surfaces through compact drawer sections instead of the old explanation-heavy capability console.

## Phases

### Phase 19: Graph-First Demo Shell

**Goal**: Replace the current three-column capability-console shell with a graph-first showcase layout led by a host-level menu.
**Depends on**: v1.2 shipped baseline
**Plans**: 3 plans

Plans:

- [x] 19-01: Rebuild the demo shell around a host-level top menu and one main graph surface
- [x] 19-02: Add compact host-pane state and a right-side overlay drawer
- [x] 19-03: Lock the new graph-first shell with focused headless regressions

**Details:**
Phase 19 removed the permanent three-column explanation shell, introduced the in-window host menu plus compact right-side `SplitView`, and locked the graph-first baseline behind focused tests.

### Phase 20: Host Menu Control Consolidation

**Goal**: Consolidate view, behavior, and runtime-facing demo controls into compact host-level menu groups that act on the same live graph session.
**Depends on**: Phase 19
**Plans**: 3 plans

Plans:

- [x] 20-01: Turn the `视图` group into a live chrome control surface
- [x] 20-02: Consolidate behavior and command-permission toggles into the `行为` group
- [x] 20-03: Add runtime-facing menu/readout sections and lock the flow with focused regressions

**Details:**
Phase 20 made the host menu operational instead of navigational-only, grouped shell and behavior toggles around one retained session, and surfaced canonical runtime summaries and diagnostics without scene switching or session rebuilds.

### Phase 21: In-Context Proof And Narrative Polish

**Goal**: Replace heavy explanation panels with compact in-context proof, live configuration summaries, and a clearer SDK narrative.
**Depends on**: Phase 20
**Plans**: 3 plans

Plans:

- [x] 21-01: Replace generic shell copy with proof-focused labels and badges
- [x] 21-02: Split runtime/proof content into dedicated live configuration and signal sections
- [x] 21-03: Align README and final regressions with the graph-first proof narrative

**Details:**
Phase 21 replaced panel-oriented copy with proof-oriented cues, added dedicated configuration/runtime/proof sections backed by the existing host state and `Editor.Session` diagnostics path, and aligned the README with the shipped demo story.

## Milestone Summary

**Decimal Phases:**

- None

**Key Decisions:**

- Lead the next UX cycle with a graph-first host menu rather than more runtime features.
- Keep one live retained editor/session path in the demo while all host adjustments act in place.
- Keep secondary showcase content behind a compact drawer instead of restoring permanent explanation rails.
- Distinguish host-owned seams from shared runtime state through short labels and live values rather than paragraph-heavy cards.
- Align the repo-facing README with the same host-menu and one-live-session proof story expressed in the UI.

**Issues Resolved:**

- The demo no longer buries the graph behind a capability-console layout.
- Host controls are no longer fragmented across unrelated explanation cards.
- Runtime proof now lives next to the current graph/session instead of in a diagnostics-heavy side layout.
- Demo-facing documentation now matches the shipped UI narrative.

**Issues Deferred:**

- Named showcase presets for common host narratives
- Guided proof tours or walkthrough sequencing
- Returning to plugin loading and richer automation implementation on top of the shipped kernel/runtime seams
- The known retained history/save semantic mismatch (`STATE_HISTORY_OK`) that predates this milestone

**Technical Debt Incurred:**

- The demo still uses the retained `GraphEditorViewModel` compatibility path as its live host surface.
- Existing nullable warnings in kernel/adapter code paths remain unresolved.
- No standalone v1.3 milestone audit file was created before archival; archive proceeded from complete phase summaries and fresh verification artifacts.

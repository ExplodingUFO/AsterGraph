<action>
- Implement plugin-loading during canonical factory resolution so both `CreateSession(...)` and `Create(...)` consume the same plugin composition result.
- Support the first explicit assembly-path plugin load baseline with a custom `AssemblyLoadContext` plus `AssemblyDependencyResolver`. Keep shared `AsterGraph.*` contract/runtime assemblies in the default context to preserve cross-boundary type identity.
- Allow multiple plugin registrations in one host composition call. Ordinary plugin load failures should surface as recoverable diagnostics or explicit load-report data rather than crashing the host process.
- Extend runtime descriptor support so the canonical session surface can advertise loader availability with a stable feature identifier such as `integration.plugin-loader`.
- Keep actual application of plugin-contributed menus, presentation, localization, diagnostics, or node providers deferred. This plan should prove loading and canonical visibility only.
- Add a focused test plugin fixture assembly plus loading tests that verify the public factory/options path can load a plugin assembly without relying on internal host reflection.
</action>

# 458.2 Avalonia Command Surface Audit

## Scope

This audit covers Avalonia input routing and workbench command projection: keyboard and pointer handlers, hosted actions, menus, toolbars, palettes, authoring tools, and view composition. It does not cover kernel command implementation details except where Avalonia consumes session descriptors.

## Current Avalonia Ownership

- `src/AsterGraph.Avalonia/Controls/GraphEditorView.axaml.cs` owns full-shell chrome composition, header actions, command palette, stencil library, fragment/export panels, validation chrome, and command-surface refresh.
- `src/AsterGraph.Avalonia/Controls/GraphEditorView.AuthoringTools.cs` owns the authoring tools panel for selection, node, and connection actions. It uses session tool descriptors and command descriptors as the source of truth.
- `src/AsterGraph.Avalonia/Hosting/AsterGraphHostedActionFactory.cs` converts session command descriptors and tool descriptors into hosted Avalonia action descriptors.
- `src/AsterGraph.Avalonia/Controls/Internal/GraphEditorDefaultCommandShortcutRouter.cs` handles descriptor-backed shortcuts and hosted action shortcuts.
- `src/AsterGraph.Avalonia/Controls/NodeCanvas.axaml.cs` owns standalone canvas pointer and shortcut event wiring.
- `src/AsterGraph.Avalonia/Controls/Internal/Interaction/NodeCanvasPointerInteractionCoordinator.cs` owns pointer routing for canvas panning, selection, drag, resize, group, and connection interactions.

## Input Routing

Keyboard:

- `GraphEditorView` registers `KeyDownEvent` in its constructor and routes full-shell shortcuts through `HandleKeyDown`.
- `GraphEditorDefaultCommandShortcutRouter.TryHandle(...)` reads `session.Queries.GetCommandDescriptors()`, applies `AsterGraphCommandShortcutPolicy`, skips input controls unless allowed, and calls `session.Commands.TryExecuteCommand(...)` for enabled descriptors.
- `NodeCanvas.axaml.cs:485` routes standalone canvas shortcuts through the same shortcut router.
- `AsterGraphCommandShortcutPolicy` allows host override/disable behavior without changing command descriptors.

Pointer:

- `NodeCanvas.axaml.cs` wires `PointerPressed`, `PointerMoved`, `PointerReleased`, `PointerWheelChanged`, and pointer capture loss.
- `NodeCanvasPointerInteractionCoordinator` uses `GraphEditorPointerInputRouter` and the interaction session to route panning, drag, resize, marquee-style selection, group operations, and connection interactions.
- Pointer interactions execute typed view model/session methods today; Phase 459 should not force every pointer gesture through string command invocations unless it is an explicit semantic command boundary.

## Command Projection Surfaces

Header toolbar:

- `GraphEditorView.axaml.cs` has a local `HeaderCommandIds` list for common actions such as save, load, undo, redo, viewport, and delete.
- Header actions are created through `AsterGraphHostedActionFactory.CreateCommandActions(...)` and shortcut policy projection.

Command palette:

- The command palette builds from `GetCommandSurfaceProjection()` and groups hosted actions by descriptor group/title.
- It includes command descriptors, authoring tool actions, and a host shell command for opening the palette.
- Recent action state is local UI state, while execution remains descriptor-backed.

Authoring tools:

- `GraphEditorView.AuthoringTools.cs` projects selection, node, and connection actions from `Editor.Session.Queries.GetToolDescriptors(...)`.
- Connection route editing uses command descriptors for `connections.route-vertex.insert`, `connections.route-vertex.move`, and `connections.route-vertex.remove` before invoking `TryExecuteCommand(...)`.
- Some workbench-only actions, such as fragment-template import/delete buttons, are created as host actions with `GraphEditorCommandSourceKind.Host`.

Menus:

- Runtime menu descriptors are built by `GraphEditorSessionMenus` and `GraphEditorSessionStockMenuDescriptorBuilder`.
- Avalonia context menu presentation adapts canonical `GraphEditorMenuItemDescriptorSnapshot` into controls.
- Menu enabled state and disabled reasons mostly come from command descriptors.

## Stock Versus Host Ownership

Stock package ownership:

- Stable command IDs, descriptor metadata, enabled/disabled state, recovery hints, command execution, menu/tool descriptors, and session command boundaries.

Avalonia ownership:

- UI layout, keyboard event routing, pointer gestures, command palette filtering/recent state, button construction, focus restoration, shortcut policy application, and presentation of disabled/recovery text.

Host/Demo ownership:

- Optional hosted actions, proof scenarios, cookbook selection, and sample-specific menu groups. Demo should remain a proof surface, not a command runtime owner.

## Phase 459 Seams

Recommended narrow write areas:

- Replace local hard-coded command lists such as `HeaderCommandIds` with a registry-backed projection or registry query that still allows the view to choose placement.
- Keep `AsterGraphHostedActionFactory` as the bridge from command descriptors to Avalonia actions; extend it only for registry/keybinding metadata needed by UI.
- Keep `GraphEditorDefaultCommandShortcutRouter` descriptor-backed, then add keybinding conflict detection outside the event handler.
- Add tests that prove header, command palette, authoring tools, and standalone canvas shortcuts consume the same registry metadata.
- Keep pointer gestures on typed interaction coordinators unless a gesture represents a semantic command already covered by `TryExecuteCommand(...)`.

## Risks

- `GraphEditorView.axaml.cs` is already a large shell file. Adding registry building, conflict detection, and command grouping directly there would create god UI code.
- Header toolbar command placement is currently local to Avalonia. If Phase 459 only updates runtime descriptors, stock header actions can still drift from supported command groups.
- Shortcut policy is projection-time behavior. Conflict detection must account for policy overrides, not just descriptor default shortcuts.
- Hosted action IDs and command IDs can differ for tool actions. Tests should verify both action identity and underlying invocation command ID.
- Some UI-only actions are legitimate host actions. Phase 459 should classify them, not force them into kernel commands.

# Phase 462.1 Audit: Selection Transform And Spatial Editing

## Goal

Phase 462 should make selection transform and spatial editing source-backed, queryable, and reusable by Avalonia controls. It should extend the existing command/session spine instead of adding a second interaction runtime.

## Current Evidence

- Selection state is node/connection-id based and filtered against the active scope in `GraphEditorKernelSelectionCoordinator.SetSelection`; it clears connection selection when node selection changes and raises `SelectionChanged`.
- `GraphEditorSessionQueries.GetSelectionSnapshot`, `GetNodePositions`, `GetNodeSurfaceSnapshots`, `GetNodeGroupSnapshots`, and `GetSelectedNodeConnectionIds` already expose the source-backed ingredients needed for transform projection.
- Node movement and layout application route through `GraphEditorKernel.SetNodePositions` and `TryApplyLayoutPlan`; both end in document mutations and `LayoutChanged` history/dirtiness.
- Node and group resizing already have separate kernel operations: `TrySetNodeSize`, `TrySetNodeGroupPosition`, and `TrySetNodeGroupSize`.
- Selection alignment/distribution is centralized in `NodeSelectionLayoutService`, but it only returns final positions for existing layout operations.
- Existing command ids include `nodes.move`, `nodes.resize`, `groups.move`, `groups.resize`, `layout.snap-selection`, and `layout.snap-all` in `GraphEditorCommandDescriptorCatalog`.
- `GraphEditorKernelCommandRouterHost` already exposes transform-adjacent host operations: selected counts, node positions, layout apply, selected snap, all-node snap, group move/resize, and viewport commands.
- Avalonia authoring surfaces currently rebuild from command descriptors and selection snapshots in `GraphEditorView.RefreshCommandSurfaceState`; selection toolbar actions are projected from command/tool descriptors, not a transform-specific state.

## Gaps

- There is no single query contract that describes a selection transform box, selected item bounds, allowed operations, rejected operations, or drag/nudge preview.
- Marquee/lasso-style selection projection is not represented as a reusable package query. Current selection can be set, but hit-test/rectangle projection is UI-local or absent.
- Keyboard nudging and constrained movement are not exposed as first-class commands with explicit delta/axis/grid semantics.
- Snap commands mutate immediately; there is no preview/snap-guide projection that Avalonia can render before commit.
- Group and route constraints are enforced by existing mutations, but Phase 462 needs proof that multi-item transform previews and commits preserve group membership, route geometry, and rejection behavior.

## Recommended Bead Boundaries

### 462.2: Source-backed selection transform contracts

Write scope:
- `src/AsterGraph.Editor/Runtime/IGraphEditorQueries.cs`
- `src/AsterGraph.Editor/Runtime/IGraphEditorCommands.cs`
- `src/AsterGraph.Editor/Runtime/Queries/GraphEditorSessionQueries.cs`
- `src/AsterGraph.Editor/Runtime/Commands/GraphEditorSessionCommands.cs`
- New runtime snapshots for selection transform bounds, transform query, transform operation, and rejection reasons.
- Kernel support near existing selection/layout mutation paths.
- `tests/AsterGraph.Editor.Tests/*SelectionTransform*`

Expected work:
- Add queryable transform snapshot from current selection and active scope.
- Add rectangle selection projection that returns stable node/group/connection references without mutating.
- Add nudge/constrained move commands that commit through existing node/group mutation paths and command events.
- Keep all calculations in Editor package; Avalonia should consume snapshots only.

### 462.3: Snap guides and Avalonia interaction projection

Write scope:
- New runtime snap guide snapshots/query contracts.
- Command descriptor/router metadata for preview/commit if needed.
- `src/AsterGraph.Avalonia/Controls/GraphEditorView*` only where it consumes shared state.
- Focused tests for snap projection and command surface visibility.

Expected work:
- Add snap-guide projection for selected nodes/groups against grid and nearby scene anchors.
- Wire stock Avalonia interaction surfaces to shared transform/snap state without duplicating layout rules.
- Keep existing `layout.snap-selection` and `layout.snap-all` as commit commands.

## Constraints

- Do not add a compatibility layer, fallback mode, scripting system, or separate UI-only transform engine.
- Do not put transform logic into Demo/Cookbook.
- Do not make Avalonia the source of truth for transform state.
- Keep implementation narrow: snapshots and operations in Editor, rendering/interaction consumption in Avalonia.

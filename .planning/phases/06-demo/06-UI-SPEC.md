## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline metadata separators, very tight label/icon gaps |
| sm | 8px | Compact badges, small control clusters, proof-row spacing |
| md | 16px | Default control spacing, normal card padding, stack rhythm |
| lg | 24px | Column section padding, major card padding, region gaps |
| xl | 32px | Showcase section separation inside a column |
| 2xl | 48px | Major visual breaks between architecture groups |
| 3xl | 64px | Reserved for page-scale breathing room only |

Exceptions: compact preview frames must use 8px or 16px inner padding depending on density; minimum interactive hit target is 44px high for primary toggles, buttons, and selectable showcase rows.

## Phase-Specific Layout Contract

Source: `F:\CodeProjects\DotnetCore\avalonia-node-map\.planning\phases\06-demo\06-CONTEXT.md:15-39`, `F:\CodeProjects\DotnetCore\avalonia-node-map\src\AsterGraph.Demo\Views\MainWindow.axaml`, `F:\CodeProjects\DotnetCore\avalonia-node-map\src\AsterGraph.Avalonia\Controls\GraphEditorView.axaml`.

### Window Composition
- Use a fixed three-column desktop layout at the `MainWindow` level.
- Recommended column proportions: left 280px, center `*`, right 360px.
- Preserve a minimum usable center width of 720px before any column begins collapsing.
- If width becomes constrained, side columns scroll vertically; the center editor remains visible and above the fold.
- Do not convert the experience into tabs, routes, or multi-page mode switches.

### Left Rail Contract
The left rail is for control and navigation only. It contains, in order:
1. Demo title and one-sentence SDK positioning summary.
2. Capability navigation for the four required architecture areas.
3. Host/runtime toggles already supported by `MainWindowViewModel`.
4. A compact legend that explains what is live, what is stock, and what is custom.

Rules:
- Navigation items are selectable card rows, not plain text links.
- Only one capability section is highlighted at a time.
- Toggle groups are visually separated from capability navigation.
- Do not place long-form explanation paragraphs in the left rail.

### Center Rail Contract
The center rail is the primary experiential surface.
- It contains one real interactive `GraphEditorView` instance as the dominant element.
- It directly demonstrates the value of the default full shell.
- Below or beside the editor, allow only compact proof companions: one shell-vs-canvas strip, one bounded standalone preview row, or one focused runtime summary strip.

Rules:
- Do not place multiple equal-weight live editors in the center area.
- The center editor must always read as the main artifact of the page.
- Any compact preview must visually read as subordinate to the main editor.

### Right Rail Contract
The right rail is for explanation, evidence, and runtime state. It contains:
1. Architecture explanation card for the currently selected capability.
2. Host connection card naming the real API/type seam.
3. Runtime/diagnostics evidence card using real session data.
4. Optional standalone/custom preview cards tied to the same editor/session.

Rules:
- Every section answers what layer this belongs to, what can be replaced/embedded, and how a host reaches it.
- Every proof card binds to real runtime objects or snapshots, not hard-coded placeholder values.
- Status bar text alone does not satisfy runtime/diagnostics proof.

## Phase-Specific Interaction Contract

Source: `F:\CodeProjects\DotnetCore\avalonia-node-map\.planning\phases\06-demo\06-CONTEXT.md:15-39`, `F:\CodeProjects\DotnetCore\avalonia-node-map\tools\AsterGraph.HostSample\Program.cs`, `F:\CodeProjects\DotnetCore\avalonia-node-map\src\AsterGraph.Avalonia\Controls\GraphEditorViewChromeMode.cs`.

### Capability Navigation
- The four required capability areas are always visible: 完整壳层, 独立表面, 可替换呈现, 运行时与诊断.
- Clicking a capability row updates the right-rail explanation and compact proof area without reconstructing the primary editor session.
- Selected state uses accent border/background treatment plus a visible Chinese label; hover uses only subtle secondary elevation or border change.

### Full Shell Proof
- Show `GraphEditorView` as the baseline integrated experience.
- Include at least one compact proof of `GraphEditorViewChromeMode` to explain full shell vs canvas-only boundary.
- Explicitly state that chrome visibility changes do not rebuild the editor runtime.

### Standalone Surfaces Proof
- Show visible proof of standalone canvas, inspector, and mini map boundaries.
- These may be miniature live surfaces or bordered preview cards, but all remain visually subordinate to the main editor.
- Each preview states that it shares the same editor/session as the center experience.
- Name the public entry points when teaching the host story: `AsterGraphCanvasViewFactory`, `AsterGraphInspectorViewFactory`, `AsterGraphMiniMapViewFactory`.

### Replaceable Presentation Proof
- Show stock vs custom presenter as a visual swap, not as two different behavior systems.
- Explicitly state which presentation seam is being replaced: node, context menu, inspector, or mini map.
- Keep the interaction explanation anchored to `AsterGraphPresentationOptions` and the existing presenter seams.

### Runtime And Diagnostics Proof
- Show session/capability/inspection information from real runtime APIs.
- Show recent diagnostics as a short, severity-aware list.
- Reserve the right rail for diagnostics history and inspection summary; do not overload the bottom status area.
- Distinguish machine-readable diagnostics from human status text.

### Localization Contract
- New Demo shell content ships Chinese-first.
- Existing English literals in `GraphEditorView.axaml` are translated where they are user-facing UX copy.
- `GraphEditorViewModel` captions resolve through `IGraphLocalizationProvider` where keys already exist.
- Technical identifiers may remain English when they help the host understand the real API seam.

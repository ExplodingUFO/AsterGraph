<objective>
Implement the first descriptor-first automation runner so hosts can execute multi-step session workflows through stable command IDs, batch them intentionally, and observe typed runtime progress/failure signals.
</objective>

<scope>

- Reuse mutation scopes for batched execution.
- Publish typed automation lifecycle/progress/failure telemetry through runtime events and diagnostics.
</scope>

<action>
- Implement synchronous automation execution on top of the canonical session path so each step uses stable command invocations instead of retained-facade methods.
- Support intentional batch execution by reusing `BeginMutation(...)` with a stable automation label rather than inventing a second transaction primitive.
- Return a machine-readable execution result that includes step outcomes plus canonical query/inspection snapshots sufficient for headless hosts to reason about the run.
- Add automation-specific typed events for lifecycle/progress/failure visibility without changing the semantics of existing generic command batching.
- Publish machine-readable diagnostics for automation start, recoverable step failure, and completion so non-event consumers still have stable telemetry.
- Add focused execution tests that prove successful multi-step automation, batch-label propagation, unknown/disabled command handling, and typed telemetry continuity.
</action>

<acceptance_criteria>
- hosts can execute a multi-step automation run from the canonical session boundary without calling `GraphEditorViewModel`
- automation runs can batch ordinary runtime notifications through mutation scope while still exposing explicit automation lifecycle/progress telemetry
- hosts receive typed automation events and machine-readable diagnostics suitable for headless or non-Avalonia consumers
- execution results expose stable machine-readable state snapshots rather than only UI text
- the first baseline remains synchronous, in-process, and descriptor-first rather than becoming a workflow engine
</acceptance_criteria>

# Phase 460.1 Command And Clipboard Audit

Bead: `avalonia-node-map-48w.3.1`

Scope: read-only audit of the current semantic editing, command routing, mutation, clipboard, and fragment paths. This artifact is the only planned write for 460.1.

## Current Behavior Map

### Session command surface

- Runtime command IDs are declared in `GraphEditorSessionCommands.KernelCommandIds`.
- Direct command methods publish `CommandExecuted` through `PublishCommandExecuted(...)`; commands returning `bool` publish only when the host mutation succeeds.
- Generic `TryExecuteCommand(GraphEditorCommandInvocationSnapshot)` delegates to `_host.TryExecuteCommand(...)` for stock commands and publishes the invoked command ID when the router returns `true`.
- `GraphEditorCommandExecutedEventArgs` carries command ID, optional mutation label, mutation-scope flag, and current status message.
- Registry projection is descriptor driven through `GraphEditorCommandRegistry.Create(...)`; every descriptor gets command-route and command-palette placement, and selected stock commands also get menu, tool, workbench, and shortcut placements.

### Copy, paste, export, and import

- `GraphEditorKernelClipboardCoordinator.TryCopySelectionAsync(...)` checks clipboard permission, creates a `GraphSelectionFragment` from selected nodes, stores it in the in-memory `GraphSelectionClipboard`, and writes serialized JSON through `IGraphTextClipboardBridge` when available.
- Fragment creation includes selected nodes, internal connections whose source and target are both selected, a top-left origin, and the primary selected node ID. It does not include external connections to nodes outside the copied set.
- `TryPasteSelectionAsync(...)` checks paste permission, prefers system clipboard text when a bridge exists and deserialization succeeds, then falls back to the in-memory clipboard.
- `TryPasteFragment(...)` checks node-create permission and connection-create permission when the fragment contains connections. It allocates fresh node IDs and connection IDs, offsets node positions from fragment origin to the next paste origin, remaps internal connection endpoints, appends pasted nodes/connections to the active scope, selects pasted nodes, and marks one `FragmentPasted` history boundary.
- `GraphEditorKernelFragmentStorageCoordinator` reuses the clipboard coordinator's fragment creation and paste operation for workspace fragment export/import and template import. Export raises `FragmentExported`; import/template import raises `FragmentImported` after successful paste.
- Fragment template export/import is gated by template-management permission plus fragment library enablement. Workspace fragment import additionally requires the file to exist.
- Compatibility tests already verify versioned clipboard/fragment payload write and legacy payload read through `EditorClipboardAndFragmentCompatibilityTests`.

### Duplicate

- `GraphEditorKernelNodeMutationCoordinator.DuplicateNode(...)` checks `Nodes.AllowDuplicate`, finds the node, calls `GraphEditorKernelDocumentMutator.DuplicateNode(...)`, gives the duplicate a new ID, offsets position by 48 px, clears `Surface.GroupId`, appends it to the active scope, selects only the duplicate, and marks a `NodesAdded` history boundary.
- The current duplicate path duplicates one node only. It does not copy selected internal connections, group membership, composite child graph content, or connection routes.

### Delete and delete-with-reconnect

- `DeleteSelection()` checks node delete permission, requires selected nodes, calls `GraphEditorKernelDocumentMutator.DeleteSelection(...)`, removes all incident connections, removes deleted node IDs from groups, cancels pending connection, clears node selection, and marks one `NodesRemoved` history boundary.
- `DeleteNodeById(...)` follows the same side-effect connection rule for one node and preserves remaining selection.
- Deleting connected nodes is blocked unless connection delete or disconnect permission is available through `CanRemoveConnectionsAsSideEffect()`.
- `TryDeleteSelectionAndReconnect()` and `TryDetachSelectionFromConnections()` share `TryReconnectSelectedMiddleNode(...)`. They require exactly one selected node, connection create/delete permission, exactly one incoming and one outgoing connection, resolvable endpoint types, compatible predecessor-to-successor ports, and no existing duplicate predecessor-to-successor connection.
- Delete-with-reconnect removes the selected node from nodes and groups, removes the two original connections, adds one new connection, clears selection, and marks `NodesRemoved`.
- Detach-with-reconnect keeps the selected node, removes the two original connections, adds one new connection, keeps the node selected, and marks `ConnectionsChanged`.
- `GraphEditorDeleteReconnectDetachTests` cover delete, detach, conflict/no-mutation, and undo as a single action.

### Reconnect and connection editing

- `TryReconnectConnection(...)` checks disconnect and create permissions, resolves the existing connection and its source output, deletes the original connection, sets a pending connection from the original source output, and marks a `ConnectionsChanged` history boundary. The next target is chosen through the normal pending connection completion path.
- `StartConnection(...)` sets or toggles pending connection from a valid output port.
- `CompleteConnection(...)` resolves port or parameter targets, checks stable type IDs, compatibility, duplicate connection, incoming replacement permission, source max connections, target max connections, then appends a new connection and clears pending state.
- `DisconnectConnection(...)`, `BreakConnectionsForPort(...)`, `DisconnectIncoming(...)`, `DisconnectOutgoing(...)`, and `DisconnectAll(...)` remove connections through the document mutator and mark connection-change boundaries.
- Connection route, label, and note edits are already modeled in `GraphEditorKernelDocumentMutator` and routed through the command router, but they are not the clipboard model's primary path.

### Command routing

- `GraphEditorKernelCommandRouter.GetCommandDescriptors()` computes availability from host permissions and state: selection count, connection count, group count, pending connection, viewport size, workspace existence, fragment existence, and clipboard coordinator capability.
- `TryExecuteCommand(...)` blocks disabled commands first, parses string arguments, and calls the typed host methods. Relevant existing IDs include `selection.delete`, `clipboard.copy`, `clipboard.paste`, `fragments.export-selection`, `fragments.import`, `nodes.duplicate`, `connections.reconnect`, `connections.connect`, and route-vertex commands.
- Router `clipboard.copy` and `clipboard.paste` currently fire-and-forget async host tasks (`_ = ...`) and return `true` after descriptor gating. Direct session methods await and publish only on success.
- The stock registry currently has placements for selection delete/export, node duplicate/delete, connection reconnect/disconnect, and many connection commands. It does not yet expose stock placements for copy/paste as first-class selection/canvas actions.

### Mutation and undo boundaries

- Kernel mutations update the active scope through `ApplyActiveScopeDocument(...)`, which merges active-scope contents back into the full document, normalizes group state, and normalizes selection/pending state.
- `MarkDirty(...)` delegates to `GraphEditorKernelHistoryCoordinator.MarkDirty(...)`, which pushes a history state and raises `DocumentChanged`.
- Undo/redo restore document, selected node IDs, primary selected node ID, active graph ID, clear connection selection, clear pending connection, raise selection/pending/document events, and use `Undo`/`Redo` document change kinds.
- Runtime mutation batching is session-side. `PublishCommandExecuted(...)` batches command executed events when a mutation scope is active; host `DocumentChanged`, `SelectionChanged`, fragment events, and pending-connection events are also buffered by session mutation plumbing.

## Risks And Invariant Checks

- Nodes: pasted or duplicated nodes must get unique IDs, preserve definition IDs, parameter values, surface state where intended, and avoid stale group IDs.
- Ports: connection endpoints must still refer to ports or parameter targets that exist on the remapped node; target kind must survive copy/paste.
- Groups: delete paths remove node IDs from groups and normalize bounds; paste currently does not copy groups. If group-aware paste is added, group IDs, node membership, bounds, collapsed state, and node `Surface.GroupId` must be remapped together.
- Connections: copied connections are internal-only; paste must never reconnect to external nodes by stale IDs. Route vertices, note text, labels, conversion IDs, and target kind need explicit preservation tests when included.
- Selection: paste selects pasted nodes and remaps primary node. Delete clears or trims selection. Undo currently restores node selection but clears connection selection.
- Pending connection: node delete cancels pending connection; reconnect sets pending connection and records a history boundary before the replacement target is chosen.
- Mutation events: every document mutation should raise one coherent `DocumentChanged` with meaningful changed node/connection IDs. Fragment export/import events should remain separate from document mutation events.
- Command executed events: direct async copy/paste methods publish only on success, but generic router copy/paste currently returns before async completion and the session publishes command execution immediately. If 460.2 formalizes these contracts, this mismatch needs either an explicit documented limitation or a synchronous/async command route correction.
- Permissions: delete-side-effect operations rely on connection delete or disconnect permission. Delete-with-reconnect currently requires connection delete, not disconnect-only permission.
- Active scope: clipboard, fragment, duplicate, delete, and reconnect operate on the active scope snapshot and merge back into the full scoped document. Composite child graph scenarios need explicit regression coverage.
- Compatibility: reconnect and connect enforce type compatibility; paste currently trusts serialized connections and does not re-evaluate compatibility.

## Handoff Notes For 460.2

- Treat the command registry as the source of discovery. Do not add local Demo command lists.
- Close the direct-method versus generic-router copy/paste event mismatch before claiming semantic command contracts are stable.
- Add command descriptors and placements at the same time as any new command IDs; keep descriptor disabled reasons/recovery hints testable.
- Batch edit commands should report one mutation boundary and one command executed event per user-facing action.
- Keep command arguments typed at the session/host boundary where possible; router string parsing should remain only the invocation adapter.

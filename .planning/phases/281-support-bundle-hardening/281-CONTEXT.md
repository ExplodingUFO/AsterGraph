# Phase 281: Support Bundle Hardening And Issue Template Activation - Context

**Gathered:** 2026-04-25
**Status:** Ready for planning

<domain>
## Phase Boundary

Make the support bundle capture enough context for first-line triage; activate the GitHub issue template for structured adopter reports.

Success criteria:
1. A support bundle generated by any host contains environment, metrics, reproduction steps, and parameter snapshots sufficient for classification without follow-up questions.
2. The GitHub issue template actively asks for the support bundle and host integration context (framework version, adapter, custom node usage).
3. ConsumerSample proof tests validate that support bundles meet the new completeness criteria.

</domain>

<decisions>
## Implementation Decisions

### Support Bundle Schema Extension
- Extend `ConsumerSampleProofResult` with optional fields (defaults preserve backward compatibility):
  - `NodeCount`, `ConnectionCount` — graph summary for triage classification
  - `FeatureDescriptorIds` — list of available capabilities to detect missing modules
  - `RecentDiagnosticCodes` — recent diagnostics to catch silent failures
- Extend `ConsumerSampleSupportBundleDocument` with `GraphSummary`, `FeatureDescriptors`, `RecentDiagnostics`
- Schema version stays at 1 (additive changes only)

### Issue Template Updates
- `adoption_feedback.yml`: add `.NET SDK version`, `UI adapter`, `Custom nodes/ports/edges` fields
- `bug_report.md`: sync environment section with adapter and custom node questions

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `ConsumerSampleSupportBundle.WriteProofBundle` — existing bundle writer
- `ConsumerSampleProofResult` — immutable record; optional params allow additive extension
- `ConsumerSampleProof.Run()` — has access to `host.Session` for graph/diagnostics queries

### Established Patterns
- Proof markers: `KEY:True/False` format
- Support bundle: camelCase JSON with schema version
- Issue templates: GitHub form schema (YAML) + markdown

### Integration Points
- `ConsumerSampleProofTests.CanWriteSupportBundleContract` validates full bundle schema
- `Program.cs` calls `WriteProofBundle` after proof run

</code_context>

<specifics>
## Specific Ideas

- Keep schema version at 1 because changes are purely additive; bump only on breaking changes.
- Feature descriptors filtered to `IsAvailable` only — unavailable features are noise for triage.
- Diagnostics limited to 16 recent entries and deduplicated to keep bundle size bounded.

</specifics>

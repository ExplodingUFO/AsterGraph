## Viewport, Render, And Hit-Test Lifecycle

1. Size and viewport state originate in the retained editor/session path. `NodeCanvas.ArrangeOverride(...)` sends the arranged size to `UpdateViewportSize(...)`; wheel, pan, minimap, focus, fit, and command routes mutate the same viewport snapshot.
2. Viewport math stays in `AsterGraph.Editor`. Avalonia calls `GraphEditorViewModel.ScreenToWorld(...)`, `PanBy(...)`, and `ZoomAt(...)` rather than owning separate viewport state.
3. `NodeCanvasViewModelObserver` reacts to `Zoom/PanX/PanY` by calling `NodeCanvasSceneHost.UpdateViewportTransform(...)`, which recomputes visible-scene projection, replaces the scene `TransformGroup`, and invalidates `GridBackground`.
4. Node/group visual lifecycle is still retained-control based. `RebuildScene(...)` clears and recreates all active-scope visual controls. Active-scope filtering is applied, but viewport projection does not yet cull node/group visual creation.
5. Connection lifecycle is redraw-based. `RenderConnections(...)` clears `ConnectionLayer.Children` and recreates visual paths/chips. During pending-connection movement, pointer position is kept in `NodeCanvasInteractionSession` and converted through `ScreenToWorld(...)`.
6. Hit testing is split: generic pointer press routing and resize edge math live in `AsterGraph.Editor.Scene`; Avalonia resolves real controls, pointer capture, cursor feedback, and visual updates.
7. Minimap operates as a second viewport control over the same session. The stock surface computes world bounds from all document nodes, draws all minimap nodes, draws the current viewport rectangle through `ViewportMath.ScreenToWorld(...)`, and recenters via `CenterViewAt(...)`.

## Performance And Invalidation Risks For Phase 467

- `NodeCanvasSceneHost.RebuildScene(...)` remains the largest structural cost: it clears group, node, and connection layers, recreates visual controls, then renders connections immediately and posts a second loaded-priority connection render.
- `NodeCanvasConnectionSceneRenderer.RenderConnections(...)` clears and recreates all connection visuals on each call. The visible-scene projection already reports visible connection IDs, but the renderer currently iterates `ViewModel.Connections` and filters only active-scope hierarchy.
- `RefreshVisibleSceneProjection(...)` currently creates a full document snapshot on every viewport transform and connection render. This gives proof data, but can become a hidden allocation/copy cost during pan/zoom and pending-connection pointer movement.
- `ViewportVisibleSceneProjection.Diff(...)` exists but is not wired into scene invalidation. Phase 467 can use it to prove bounded invalidation before attempting control recycling.
- `GridBackground.Render(...)` redraws line loops on every viewport invalidation. Spacing normalization bounds line density, but Phase 467 should measure redraw cadence under wheel/pan.
- `GraphMiniMap` balanced mode invalidates on every viewport change. Throughput mode avoids viewport subscriptions and refreshes viewport only when rendered, so Phase 467 needs explicit cadence proof to prevent a stale rectangle perception during continuous pan/zoom.
- `GraphMiniMap.CreateProjection(...)` enumerates every document node when projection is rebuilt. Lightweight mode caches this, but document/selection changes still rebuild all minimap nodes.
- Pointer drag paths can trigger high-frequency updates: node drag applies offsets, updates group drop zones twice around auto-expand checks, updates group visuals when hover state changes, and redraws pending connections if present.
- Resize feedback hover walks visual surface maps and translates pointer coordinates. Existing logic is scoped to registered node/group surfaces, but Phase 467 should prove hover hit-test latency on dense visible canvases.

## Missing Proof Points

- No current test proves `NodeCanvasSceneHost` uses `ViewportVisibleSceneProjection` to cull or bound actual Avalonia node/group/connection visual work.
- No current test measures or asserts allocation/call cadence for `CreateDocumentSnapshot()` during continuous pan/zoom or pending-connection drag.
- No current test proves connection rendering can be bounded to visible connection IDs; current renderer redraws from `ViewModel.Connections`.
- No current test measures grid redraw cost or validates grid line count under extreme zoom ranges.
- No current test proves minimap viewport rectangle freshness under throughput mode during real continuous pan/zoom; existing proof checks projection reads, not visual invalidation cadence.
- Hit-test coverage is strong for routing and resize handles, but there is no dense-canvas latency proof for hover resize feedback, selection rectangle hit filtering, or connection path hit areas.
- Existing scale-baseline docs cover runtime/session budgets, but visible Avalonia rendering budgets still need a dedicated Phase 467 proof gate.

## Handoff Notes

- Keep viewport state canonical in `AsterGraph.Editor`; do not introduce Avalonia-owned viewport state.
- Keep Phase 467 incremental. The current retained-control renderer can be tightened with projection, invalidation, and measurement before considering larger rendering changes.
- Preserve pending-connection and resize-preview behavior when bounding connection rendering; those paths deliberately use live anchors and pointer screen position instead of stale geometry snapshots.
- Treat minimap throughput mode as a policy contract. If the viewport rectangle is allowed to lag visually during continuous pan/zoom, make that explicit in tests and docs.
- Avoid widening graph-size claims until a visible Avalonia render proof exists. Runtime/session scale proof is not the same as canvas visual throughput proof.
- Suggested integration order: viewport projection/invalidation proof first, connection render bounding second, minimap cadence third, grid/hit-test latency proof fourth, docs last.

## Command Focus Context Lifecycle Summary

1. Command state starts in the kernel. `GraphEditorKernelCommandRouter.GetCommandDescriptors` derives enabled state and disabled reasons from current document, selection, viewport, history, pending connection, and command permission state.
2. Session queries append navigation and plugin descriptors, then `GraphEditorCommandRegistry.Create` adds route, keyboard, workbench, context-menu, and tool placements. Registry metadata is static for stock placements, while descriptor enabled state is live.
3. Hosted Avalonia surfaces build `AsterGraphHostedActionDescriptor` projections from descriptors and registry placements. `GraphEditorView` refreshes this projection when document, selection, viewport, command, or pending-connection events fire.
4. Keyboard routing has two layers. `GraphEditorView.HandleKeyDown` handles command-palette shortcuts globally enough to work from input controls, then routes hosted actions while excluding input controls and `connections.cancel`. `NodeCanvas.HandleCanvasKeyDown` routes descriptor shortcuts for the focused canvas and includes pending connection cancel.
5. Focus recovery is explicit for the command palette. Opening captures the prior control and focuses the palette search box. Closing restores the captured focus target if still visible/focusable, otherwise returns focus to the canvas or the view.
6. Context menus start from Avalonia `ContextRequested`. The canvas coordinator focuses the canvas, adjusts selection for node/port targets, resolves world position, builds a `ContextMenuContext`, asks the session for canonical descriptors, and opens the configured presenter with `IGraphEditorCommands`.
7. Menu item execution re-enters the same canonical command route through `IGraphEditorCommands.TryExecuteCommand`. Disabled menu items are disabled at descriptor-build time; some disabled command invocations can also publish status from the kernel block list.
8. Navigation/focus commands are session-level commands, not kernel commands. They can select nodes/connections, enter scopes, center viewport, and publish `CommandExecuted` for focus and bookmark workflows.

## Risks And Coupling Points For Phases 468 And 469

- Phase 468 customization should treat command descriptors and canonical menu descriptors as the public contribution surface. Extending node/edge visual customization by adding direct Avalonia menu items would bypass the session command route and fragment disabled-reason/state consistency.
- Phase 468 edge/node quick actions currently couple to registry stock placement ids plus retained adapter overrides. New node/edge toolbar contributions should add descriptor/placement proof instead of expanding `GraphContextMenuBuilder`, because the canvas path is already canonical.
- Phase 468 custom node body/editor controls must respect `IGraphEditorInputScope` or equivalent focus boundaries. The shortcut router only suppresses stock shortcuts when the event source is a `TextBox`, `ComboBox`, or `IGraphEditorInputScope`.
- Phase 468 custom context menu presenters receive descriptors plus `IGraphEditorCommands`. They should avoid executing retained VM methods directly, otherwise disabled state, command events, and plugin/kernel routing diverge.
- Phase 469 layout/spatial commands should keep using command descriptors and `GraphEditorCommandInvocationSnapshot` payloads. Layout menus already expose align/distribute/snap commands through context menus, tools, and registry placements.
- Phase 469 spatial gestures need to coordinate with focus recovery. Pointer/drag/resize paths focus the canvas before interaction; new spatial handles should preserve that behavior or keyboard follow-up actions will be routed unpredictably.
- Phase 469 should verify command-surface refresh cost and freshness after layout preview/apply/cancel flows. `GraphEditorView` refreshes on broad session events; adding high-frequency layout preview events to the same path could over-refresh.
- Both phases should preserve the split between view-level hosted actions and canvas-level descriptor shortcuts. `connections.cancel` is intentionally excluded from `GraphEditorView` and included in `NodeCanvas`; widening this accidentally could make Escape close palette, cancel pending connection, or both depending on focus.

## Missing Proof Points

- No focused test was found for view-level shortcut routing suppressing destructive commands from `TextBox`, `ComboBox`, and `IGraphEditorInputScope` while still allowing the command palette shortcut from inputs.
- No focused test was found for duplicate shortcut conflict reporting after `AsterGraphCommandShortcutPolicy.ShortcutOverrides` changes effective shortcuts in a full hosted projection.
- No focused test was found for the intentional `connections.cancel` split: excluded at `GraphEditorView` action routing and included at `NodeCanvas` descriptor routing when a pending connection exists.
- No focused test was found for canonical context menus re-querying descriptor enabled state after selection/pending-connection changes immediately before menu open.
- No proof was found that custom `IGraphContextMenuPresenter` implementations cannot mutate command state outside `TryExecuteCommand`; current boundary is convention plus interface shape.
- Legacy retained `GraphContextMenuBuilder` remains tested, but the audit did not find an explicit guard ensuring new canvas context menu work cannot regress back to retained menu descriptors.

## Handoff Notes For Integration

- Keep future command/focus/context work on the canonical session route: descriptor/query -> registry/placement -> hosted action/menu projection -> `TryExecuteCommand`.
- Treat `GraphContextMenuBuilder` as retained compatibility. New canvas menu work should start from `GraphEditorSessionMenus` and `GraphEditorSessionStockMenuDescriptorBuilder`.
- Preserve the keyboard ownership split: `GraphEditorView` handles hosted shell shortcuts and command palette focus; `NodeCanvas` handles canvas descriptor shortcuts and pending connection cancel.
- Preserve focus before context interaction. Node, port, canvas, drag, resize, and pointer interaction paths currently restore canvas focus before changing selection or opening menus.
- For Phase 468, require custom node/editor controls to participate in input-scope suppression before accepting keyboard shortcuts as complete.
- For Phase 469, require layout/spatial workflows to prove both command execution and command-surface freshness after state changes, not just mutation success.

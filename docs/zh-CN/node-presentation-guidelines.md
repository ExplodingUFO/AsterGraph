# 节点呈现建议

这份文档给出 `NodePresentationState` 的推荐用法。

它的目标是统一编辑器侧的呈现习惯，而不是把宿主自己的业务语义硬塞进 AsterGraph。

## Badge 数量

推荐可见 badge 数量：

- 大多数节点保持在 `0-3` 个
- 如果节点状态超过 `3` 个，先在宿主侧聚合，再把结果发给编辑器

这样做的原因：

- badge 太多会明显降低扫读效率
- 同时出现太多状态时，节点之间会更难比较

推荐聚合方式：

- 保留优先级最高的运行时状态
- 把次级状态合成一个摘要 badge，例如 `+2`
- 额外细节放进 tooltip

## Badge 文本长度

推荐 badge 文本长度：

- 大约 `2-8` 个可见字符
- 优先使用短的操作性词汇，例如 `Ready`、`Dirty`、`Error`、`Host`

避免：

- 句子式 badge
- 已经在别处可见的长业务名词
- 只有细微措辞差异的一组 badge

如果某个状态需要更长解释，badge 保持短文本，把详细说明放进 `ToolTip`。

## 状态栏长度

推荐状态栏文本长度：

- 大约 `8-24` 个可见字符
- 足够表达一条简短的操作性总结

好的例子：

- `Preview available`
- `Waiting for input`
- `Last run failed`

避免：

- 段落式说明
- 把多个无关状态挤进同一条状态栏
- 用状态栏替代 inspector 或详细信息面板

## Tooltip 建议

Tooltip 应该补充信息，而不是重复当前已经可见的文字。

推荐结构：

1. 这个状态代表什么
2. 为什么它重要，如果这一点并不显然
3. 用户下一步该做什么，如果有必要

好的例子：

- `Host reports the node is ready for preview.`
- `Execution failed on the last run. Open details in the host inspector.`
- `Two additional host statuses were aggregated into this summary badge.`

避免：

- 原样重复 badge 文本
- 直接塞入原始日志或大段错误负载
- 在共享 UI 里暴露只有宿主内部才懂的术语，除非这些词本来就是用户界面语言

## 颜色与强调

颜色应该表达优先级，而不是纯装饰。

推荐规则：

- 一个节点状态快照只保留一种主强调色
- badge 和状态栏里同类语义使用一致的强调色
- 高饱和 warning / error 色只留给真正需要注意的状态

避免：

- 每个 badge 都用一种强烈颜色
- 同一个节点混用多套互不相关的 warning 色系
- 在短文本足够表达时仍然只靠颜色区分

具体配色仍然由宿主自己决定。

## 宿主与编辑器的职责边界

宿主负责：

- 运行时状态的业务含义
- badge 和状态栏文本
- tooltip 细节
- 从业务状态到强调色的映射

AsterGraph 负责：

- 渲染传入的状态快照
- badge 和状态栏的布局行为
- 保持编辑器契约是框架中立的

这个边界很重要：状态代表什么由宿主决定，编辑器只决定这些状态如何显示。

## 实用默认值

如果宿主还没有更强的设计系统，可以先从下面这组默认值开始：

- 大多数节点保持 `0-2` 个 badge
- 只有节点存在可操作的运行时状态时才显示一条短状态栏
- 只有在 tooltip 能补充额外信息时才提供 tooltip
- 每个快照只保留一种主强调色

这通常已经足够在保持图可读性的同时展示运行时上下文。
